文档架构地图

人给判断，文档保存判断，AI 按文档执行
输入层：问题从哪里来

用户判断 / 反馈

产品方向、体验反馈、真实机器 smoke、哪些风险能接受。

GitHub Issues / Tester 反馈

外部用户的 P0/P1 信号，进入 issue tracker 或 release readiness。

日志 / CDP / Smoke

把“感觉坏了”变成可追踪证据，避免只靠聊天记忆。

控制层：AI 从哪里领任务

exec-plans/active

唯一当前任务入口。只放正在推进的计划，其他都不该从这里领任务。

tech-debt-tracker / issue-tracker

长期风险、已知问题、用户反馈、阻断级别和解决状态。

Smoke Ledger

真实账号、真实包、真实 UI 的验证记录。证明“真的跑过”。

知识层：改代码前读什么

handover/

系统怎么工作：数据流、架构、关键入口、已有设计。

guardrails/

哪些边界不能破：Runtime、Provider、DB、权限、发布。

research / insights

事实来源和产品判断：为什么选这个方案，为什么不选那个。

理想状态下，AI 不应该“全仓乱读”。它应该先看控制层确认当前任务，再看对应知识层校准边界，最后把验证证据回写到计划或技术债。

从需求到代码的文档工作流

这就是 CodePilot 当前 Vibe Coding 的操作系统
1

Signal

用户反馈、日志、截图、CI 失败、真实包 smoke。先形成可描述的问题。

2

Triage

进入 issue tracker / active plan，判断 P0/P1/P2、范围和不做边界。

3

Context

读 handover、guardrails、research，确认历史结论和不能破的契约。

4

Implementation

Claude Code 执行代码，Codex 做方案审查、边界确认和复盘。

5

Evidence

测试、CDP、真机、Release smoke 写回 Smoke Ledger / tech debt / handover。

计划文档生命周期

让 AI 知道哪里能做，哪里只能查
Active 当前推进。AI 可以从这里领任务，但必须有用户结果、不做边界和验收方式。
Completed 已完成。保留决策日志和 smoke evidence，只能查历史，不能当新任务。
Deferred 用户明确暂缓。保留方案，但恢复前必须由用户主动重启。
Superseded 被新计划接管。仅作历史参考，顶部 Archive note 指向当前入口。

新的判断：文档系统分 6 层治理

不是所有 Markdown 都是一种文档
最急

1. 任务控制面

exec-plans/ 回答“现在做什么”。当前主要问题是 active 混入 completed / deferred / superseded。

需补触发

2. 系统交接层

handover/ 记录系统怎么工作，但需要写清“改哪些文件前必须读我”。

覆盖不均

3. 安全契约层

guardrails/ 方向正确，但 13 份里仍有 stub；看板应区分 complete / stub / missing。

去向不清

4. 调研事实层

research/ 价值高，但需要 outcome：adopted / rejected / reference only / pending。

易过期

5. 发布时效层

preview/ 天然短命；必须标清有效版本、是否 archive、是否仍对应真实 Release 资产。

保留判断

6. 产品洞察层

insights/ 不负责执行，只沉淀“为什么这件事对产品重要”和可复用判断。

看板后续不应该只统计“多少文档过时”，而要统计每一层是否有入口、状态、证据和触发条件。 这会让人和 AI 在 30 秒内知道：哪里能领任务，哪里只查历史，哪里必须先读再改代码。

各类文档的健康问题

下一版看板应持续追踪这些维度
Lifecycle

exec-plans

问题是状态污染。治理目标是 active 只放当前任务，completed / deferred / superseded 各有明确归宿。

Trigger

handover

问题不是价值低，而是触发条件不够显式。每份应写“改哪些文件/模块前必须读”。

Coverage

guardrails

13 份护栏中有稳定契约也有 stub。数量不能代表覆盖度，需要区分完整、草稿、缺失。

Outcome

research

42 份 research 应有去向：被采纳、被否决、仅参考、待决策。否则调研会变成孤岛。

Freshness

preview / release

这类文档过期最快。必须标有效版本、真实资产、是否 archive，避免用户装错包。

Judgment

insights

只记录可复用产品判断，不承担任务入口；质量看是否被 handover / plan 引用，而不是数量。
